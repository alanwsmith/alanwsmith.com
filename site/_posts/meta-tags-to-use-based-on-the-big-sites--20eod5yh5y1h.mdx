Here's my thinking:

- `<!DOCTYPE html>` 


see also: https://dev.to/reedbarger/easily-generate-meta-tags-for-your-react-app-15gd for the google stuff to look into.



Here's some notes while I figure things out:

- These are copies of the `<head>` section/data from the various pages. 
- I removed `<script>` and `<style>` tags. I also removed `<link>` tags for `text/css`. Everything else has been left in place. (Oh, yeah, I also removed `<link>` calls for fonts.)
- Twitter really doesn't have a `<head>` tag or a `<title>` tag in the source. It also doesn't have a closing `</body>` tag in the source file. All of these show up under the "Elements" tab though.
- Note if you're using react, you need to do `charSet` with a capital `S` instead of `charset` where it's lowercase. 
- It's funny to me that google starts with lowercase `<!doctype html>` instead of `<!DOCTYPE html>`. I thought it was supposed to always be uppercase. I wonder what the reason for the lowercase is. 
- While I shortened and obfuscated some values, no attributes were removed.
- I've not seen the thing that YouTube Video Pages are doing where there's a `<div>` in the head that has meta tags wrapping the `http://schema.org/VideoObject` stuff. Especially weird since it has `<span>` elements in it.
- There were a ton of `rel="preload" as="script"` tags in the Facebook Elements. I just kept one for reference.
- It surprises me that there's not explicit meta tags for open graph content in the amazon pages.
- My understanding is that the Star Wars site is a Wordpress site. Including it since there's a huge number of those out there (understanding that it almost certainly has a bunch of customiztion)
- Twitter title for the star wars content page has the `<em>` element in it. Seems like not the best idea.
- The `msapplication-` stuff on the star wars page is commented out. Makes me wonder if there's a technical reason for that.
- Star Wars has all the old `if lt IE 7` stuff. I'm guessing that's more legacy code hanging around than for a specific target to that browser. 
- I've heard both that keywords don't matter and that Bing might actually treat them like a signal for spam. For now, I'm going to use a single keyword with the category of the page when it's available.
- Note that the Star Wars page uses multiple `twitter:` and `og:` calls for the same subtag (e.g there are multiple `og:image` calls). Need to look that up.
- The Star Wars site uses `<meta property="og:type" content="article" />` for posts and `<meta property="og:type" content="website">` which is how I setup too.
- Facebook uses only a `<link rel="shortcut icon" href="https://static.xx.fbcdn.net/rsrc.php/yb/r/hLRGG_y0J.ico" />` instead of individual files. Maybe see about moving over to a single .ico file or at least compare the options a little more.

Notes:

- Going with lowercase for `<meta charset="utf-8">` based on [MDN](https://developer.mozilla.org/en-US/docs/Glossary/character_encoding)
- Prefetching note from [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-DNS-Prefetch-Control) for `x-dns-prefetch-control`. Sounds like it's not really necessary, but that's a bit unclear and it doesn't seems like it would hurt.
- I don't really need the canonical since my pages shouldn't be duplicated, but I like the link being there regardless. (e.g. `<link rel="canonical" href="https://www.alanwsmith.com/">`)
- According to [this stackoverflow answer](https://stackoverflow.com/a/4696517/102401), `<meta charset="utf-8"/>` and `<meta http-equiv="content-type" content="text/html; charset=UTF-8">` are equivalent for HTML5. Since that's what I'm going for, I'll be using the shorter one. 



Things I'm skipping for now:

```html
<meta http-equiv="x-dns-prefetch-control" content="on">
<link rel="dns-prefetch" href="https://www.mozilla.org/contribute/">
<link rel="dns-prefetch" href="//www.mozilla.org/contribute/">
```

- Everything is on my domain right now but I'll add these in if there's stuff I need to pull from other domains. Details on [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-DNS-Prefetch-Control)
